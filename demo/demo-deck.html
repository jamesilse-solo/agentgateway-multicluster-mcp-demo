Solo.io AGENTREGISTRY
Centralized Catalog

One catalog. Every agent knows where to connect.

AgentRegistry is the MCP service directory for your entire platform. Agents look up tools by name — not by URL.

  • Namespace convention — reverse-domain naming (com.amazonaws/mcp-everything-local) scopes servers to their owning team or cloud
  • Discovery API — GET /v0/servers?search=com.amazonaws returns remotes, schema refs, and metadata in one call
  • Open registration — any team posts a server entry with a URL and a JSON schema reference; the gateway automatically discovers and routes to it
  • No-login API for demo — anonymous auth enabled; production deployments use OIDC with role-based write control

Solo.io AGENTGATEWAY
Policy Enforcement Point

All MCP traffic flows through one policy enforcement point.

AgentGateway Enterprise is the single control point for authentication, rate limiting, observability, and routing across every MCP tool call.

AUTHENTICATION

OIDC / JWT Validation

Every inbound request carries a Bearer token. ExtAuth validates it against Keycloak. No valid token — no tools. Agents are first-class principals, not anonymous callers.

ROUTING

Path-based MCP routing

/mcp → cluster1 local MCP server
/mcp/remote → cluster2 via HBONE
/mcp/registry → AgentRegistry catalog

RATE LIMITING

Per-agent & global limits

Rate limits configurable per JWT identity. Protects downstream tool servers from runaway agents. Enforced at the gateway, transparent to backends.

OBSERVABILITY

Full request telemetry

Every MCP session, tool call, and token validation emits traces and metrics. The Enterprise UI surfaces request logs, latency, auth outcomes, and route activity.

GOVERNANCE

Enterprise control plane UI

AgentRegistry Enterprise UI at localhost:8080 — catalog, traffic logs, auth outcomes, deployed agents. Gloo Mesh UI at localhost:8090 — cross-cluster topology and data plane health.

AMBIENT MESH

Sidecar-free security

All workloads run under Istio ambient mesh. ztunnel provides SPIFFE/SVID identity and HBONE mTLS with zero pod changes — no proxy injection, no restarts.

Solo.io CROSS-CLUSTER FEDERATION
Multi-Cluster Routing

Cross-cluster calls are transparent to the agent.

The agent sends one request. The gateway decides whether to route it locally or across clusters — based on path, policy, or load. Agents never need cluster-specific URLs.

  • Path-based federation — /mcp/remote is a permanent route to cluster2's MCP servers; agents call the same AgentGateway LB regardless of which cluster hosts the tool
  • HBONE mTLS — the east-west gateways on both clusters handle mutual TLS over HBONE (HTTP/2 CONNECT tunnel). Traffic is encrypted and authenticated at the infrastructure layer with no application changes
  • SPIFFE/SVID workload identity — every pod has a cryptographic identity issued by the ambient mesh CA. Authorization policies use these identities, not IP addresses
  • Add clusters as spokes — onboarding a new cluster is a spoke registration in AgentGateway and a new HTTPRoute; existing agents and registry entries need no changes
  • Consistent policy — auth, rate limiting, and observability policies defined on the hub apply to all routes including cross-cluster ones

KEY MESSAGE

Federation is bidirectional and symmetric. Agents on either cluster reach tools on any cluster using the same path convention. Adding a third cluster is one new HTTPRoute.

Solo.io EXTERNAL MCP SERVERS
SaaS & Public Endpoints

External tools, same gateway. Agents don't know the difference.

An MCP server outside the cluster — public SaaS like search.solo.io/mcp, a partner API, a self-hosted service — is wired up with exactly the same two resources as an internal one. Auth, rate limits, observability still apply.

  • Path: /mcp/search on the same AGW Hub LB the agents already use
  • Outbound TLS to search.solo.io:443 — the gateway terminates the agent's connection, then originates a fresh HTTPS connection upstream
  • Single source IP to the SaaS — the AGW Hub LB's public IP. Partners can write a one-line allowlist instead of tracking pod IPs
  • Same auth + audit as internal MCP — the agent's JWT is validated, every call shows in the AGW Enterprise UI traces
  • Drops in zero extra agent code — agents discover io.solo/search-solo-io from the registry and just call
Solo.io TOOL-LEVEL RBAC
Per-Tool Access Control

Restrict what an agent can call. Not just where it can connect.

An MCP server typically exposes many tools. The gateway can hide the ones a given agent has no business calling — applied at the backend, evaluated before the upstream server ever sees the request.

  • Backend-side allowlist — declare the allowed tool names on the AgentgatewayBackend and the gateway filters them out of tools/list AND rejects tools/call for everything else
  • Defense in depth — even a prompt-injected agent can only call what's been pre-authorised
  • Per-identity RBAC — OPA Rego in ExtAuth handles the harder case ("admins can delete, others can only search") using JWT claims
  • Whole-tier isolation — the allowlist on the backend is the simple "every agent through this route sees only these tools" lever
Restrict search.solo.io to just `search` 
apiVersion: agentgateway.dev/v1alpha1
kind: AgentgatewayPolicy
metadata:
  name: search-solo-io-policy
spec:
  targetRefs:
  - group: agentgateway.dev
    kind: AgentgatewayBackend
    name: search-solo-io
  backend:
    tls:
      sni: search.solo.io
    mcp:
      authorization:
        action: Allow
        policy:
          matchExpressions:
          - 'mcp.tool.name == "search"'
Effect on the agent
tools/list → ["search"] (1 tool)
tools/call name=search → 200 OK
tools/call name=delete → 403 MCP permission error
↳ rejected at the gateway; never reaches search.solo.io
